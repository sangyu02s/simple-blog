# 交互体验优化设计文档

## 1. 背景

当前项目已经具备认证、发帖、评论、点赞等基础能力，但在实际使用过程中仍存在一些明显的交互问题：

1. 后端已经返回了较明确的校验错误信息，但前端统一用固定失败文案覆盖，导致用户无法快速知道输入问题在哪里。
2. 顶部导航同时显示“登录”和“注册”，在已登录状态下仍保留登录入口，信息重复且不够简洁。
3. 未登录状态下依然显示“发帖”入口，会带来无效点击和额外挫败感。

本轮优化目标是提升当前项目的基础可用性和表单交互体验，而不引入新的业务复杂度。

## 2. 优化目标

本轮优化包含以下 3 个方向：

### 2.1 错误提示优化
- 前端优先显示后端返回的具体错误信息
- 避免所有失败都退化成同一条笼统提示
- 让用户能直接知道“用户名太短”“摘要太短”“评论内容过短”等具体原因

### 2.2 登录与注册入口优化
- 顶部导航在未登录状态下只显示“登录”
- 注册入口放在登录页底部
- 登录入口放在注册页底部
- 已登录后顶部不再显示“登录”入口

### 2.3 未登录状态下隐藏发帖入口
- 顶部导航中未登录时不显示“发帖”
- 避免用户进入本就无法成功操作的路径

## 3. 设计方案

### 3.1 前端错误处理方案

新增统一的错误信息提取逻辑：
- 如果后端返回结构中存在 `message` 字段，则优先展示该字段
- 如果后端未返回可用信息，再回退到前端默认文案

适用范围：
- 登录
- 注册
- 发帖
- 评论
- 点赞（如后续需要）

这样可以在不改动后端接口结构的前提下，直接提升表单可理解性。

### 3.2 顶部导航展示规则

#### 未登录时显示
- 首页
- 登录
- 后台

说明：
- “注册”不再常驻在顶部导航中
- “发帖”在未登录时隐藏

#### 已登录时显示
- 首页
- 发帖
- 后台
- 退出（带当前用户昵称）

说明：
- 已登录后不再显示“登录”入口
- 继续保留后台入口，后续由权限控制决定是否可真正访问

### 3.3 登录页与注册页跳转关系

#### 登录页底部
- 文案：`还没有账号？去注册`
- 点击后跳转到注册页

#### 注册页底部
- 文案：`已有账号？去登录`
- 点击后跳转到登录页

这样既保留入口完整性，又避免顶部导航冗余。

## 4. 影响范围

### 4.1 前端文件
- `frontend/src/api/*`
- `frontend/src/stores/authStore.ts`
- `frontend/src/pages/LoginPage.tsx`
- `frontend/src/pages/RegisterPage.tsx`
- `frontend/src/pages/WritePostPage.tsx`
- `frontend/src/pages/PostDetailPage.tsx`
- `frontend/src/App.tsx`
- `frontend/src/styles.css`

### 4.2 后端文件
- 本轮原则上不需要修改后端接口结构
- 直接复用当前统一错误响应中的 `message`

## 5. 验证方案

### 5.1 错误提示验证
- 用户名太短时，前端显示后端返回的具体长度提示
- 密码太短时，前端显示具体长度提示
- 发帖摘要过短时，前端显示摘要长度校验提示
- 评论内容过短时，前端显示评论长度校验提示

### 5.2 导航显示验证
- 未登录时顶部只显示“登录”，不显示“注册”与“发帖”
- 已登录时顶部显示“发帖”和“退出”，不显示“登录”

### 5.3 页面跳转验证
- 登录页底部可以跳转到注册页
- 注册页底部可以跳转到登录页

## 6. 结论

本轮优化不改变当前业务能力边界，但会明显改善表单反馈质量和导航可理解性，使当前项目更接近可用产品形态。
