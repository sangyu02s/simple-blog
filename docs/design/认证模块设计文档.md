# 认证模块设计文档

## 1. 模块目标

认证模块用于建立 Simple Blog 的用户身份体系，为后续发帖、评论、点赞、后台管理等功能提供统一的登录能力与权限边界。

本模块首版目标包括：
- 用户注册
- 用户登录
- 获取当前登录用户信息
- 基于 JWT 的无状态认证
- 普通用户与管理员的基础角色区分

## 2. 设计范围

本次认证模块覆盖以下内容：

### 2.1 后端范围

- 用户实体映射
- 用户数据访问层
- 注册接口
- 登录接口
- 当前用户信息接口
- JWT 生成与解析
- Spring Security 鉴权配置
- 密码加密存储

### 2.2 前端范围

- 登录页表单提交
- 注册页表单提交
- 认证接口请求封装
- 登录态本地存储
- 当前用户信息拉取
- 基础错误提示与提交状态处理

## 3. 核心设计

### 3.1 认证方案

采用 **JWT + Spring Security**。

原因如下：
- 当前项目是前后端分离结构，JWT 更适合前后端分离场景
- 后端不需要维护服务端 Session
- 前端后续接入文章、评论、后台接口时，只需统一携带令牌

### 3.2 密码处理

密码不直接明文存储，使用 Spring Security 提供的 `PasswordEncoder` 进行哈希处理。

设计要求：
- 注册时对密码加密后写入 `users.password_hash`
- 登录时通过 `PasswordEncoder.matches` 校验
- 任何接口都不返回密码字段

### 3.3 角色模型

首版角色设计：
- `USER`：普通用户
- `ADMIN`：管理员

首版注册默认创建 `USER`。

### 3.4 状态模型

首版用户状态设计：
- `ACTIVE`：正常可用
- `DISABLED`：禁用

登录时只允许 `ACTIVE` 用户通过认证。

## 4. API 设计

### 4.1 注册接口

`POST /api/auth/register`

请求体：
- `username`
- `nickname`
- `password`

处理逻辑：
- 校验字段长度与非空
- 校验用户名是否重复
- 密码加密
- 创建默认角色为 `USER`、状态为 `ACTIVE` 的用户
- 返回登录结果结构，便于注册后直接进入登录态

### 4.2 登录接口

`POST /api/auth/login`

请求体：
- `username`
- `password`

处理逻辑：
- 根据用户名查询用户
- 校验用户状态是否可登录
- 校验密码是否匹配
- 生成 JWT
- 返回 token 和当前用户基础信息

### 4.3 当前用户接口

`GET /api/auth/me`

处理逻辑：
- 从认证上下文中获取当前用户
- 返回用户基础资料

## 5. 返回结构设计

认证相关接口建议统一返回：
- `token`
- `user`

其中 `user` 包含：
- `id`
- `username`
- `nickname`
- `role`
- `status`

这样前端登录成功后可以直接保存 token 和当前用户，不需要额外再做一次结构转换。

## 6. 安全设计

### 6.1 放行接口

以下接口匿名可访问：
- `POST /api/auth/register`
- `POST /api/auth/login`
- `GET /api/posts/**`（后续文章公开接口）
- `GET /api/tags/**`（后续标签公开接口）

### 6.2 需要认证的接口

以下接口必须登录：
- `GET /api/auth/me`
- 发帖接口
- 评论接口
- 点赞接口

### 6.3 管理员接口

`/api/admin/**` 后续统一要求 `ADMIN` 角色访问。

### 6.4 JWT 处理方式

后端通过过滤器读取请求头：
- `Authorization: Bearer <token>`

处理逻辑：
- 校验 token 是否有效
- 解析用户名
- 加载用户信息
- 写入 Spring Security 上下文

## 7. 前端交互设计

### 7.1 登录流程

- 用户提交用户名和密码
- 前端调用登录接口
- 成功后保存 token
- 更新当前用户信息
- 跳转到首页

### 7.2 注册流程

- 用户提交用户名、昵称、密码
- 前端调用注册接口
- 成功后保存 token
- 更新当前用户信息
- 跳转到首页

### 7.3 登录态恢复

页面初始化时：
- 从本地存储读取 token
- 若存在 token，则调用 `/api/auth/me`
- 若请求成功，恢复当前用户
- 若请求失败，清理失效 token

## 8. 数据结构设计

### 8.1 后端重点对象

- `User`：用户实体
- `UserRole`：用户角色枚举
- `UserStatus`：用户状态枚举
- `RegisterRequest`：注册请求体
- `LoginRequest`：登录请求体
- `AuthResponse`：认证响应体
- `CurrentUserResponse`：当前用户响应体

### 8.2 前端重点对象

- `AuthResponse`
- `CurrentUser`
- `LoginFormValues`
- `RegisterFormValues`
- `AuthStore`

## 9. 重点实现约束

- 所有认证相关说明、设计文档、重点注释统一使用简体中文
- 密码字段不进入响应体
- 用户不存在与密码错误使用统一错误提示，避免泄露过多账户信息
- token 存储先采用浏览器 `localStorage`，后续可根据安全策略再调整
- 首版优先保证主流程跑通，不提前引入复杂刷新令牌机制

## 10. 验证方案

### 10.1 后端验证

- 注册新用户成功
- 重复用户名注册失败
- 正确用户名密码登录成功
- 错误密码登录失败
- 未登录访问 `/api/auth/me` 失败
- 携带有效 token 访问 `/api/auth/me` 成功

### 10.2 前端验证

- 注册表单可提交并进入登录态
- 登录表单可提交并进入登录态
- 页面刷新后能恢复登录态
- token 失效时可正确清理本地状态

### 10.3 联调验证

- 前端请求头中正确携带 Bearer Token
- 后端能正确解析 token 并识别当前用户
- 注册和登录后返回的用户信息能正确展示
