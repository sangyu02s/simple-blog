# 评论模块设计文档

## 1. 模块目标

评论模块用于为文章提供基础互动能力，使博客从“可发布、可浏览”扩展到“可讨论”。

本模块首版目标包括：
- 查看文章评论列表
- 已登录用户发表评论
- 评论与文章作者关联
- 文章评论数自动维护

## 2. 设计范围

### 2.1 后端范围

- 评论实体映射
- 评论列表接口
- 发表评论接口
- 评论状态基础能力
- 文章 `commentCount` 联动更新

### 2.2 前端范围

- 文章详情页评论列表展示
- 文章详情页评论输入与提交
- 未登录时评论拦截提示
- 评论提交后刷新评论列表与文章评论数

## 3. 核心设计

### 3.1 评论状态设计

首版评论状态：
- `VISIBLE`：可见
- `HIDDEN`：隐藏

当前阶段只读取 `VISIBLE` 评论。

### 3.2 评论归属规则

发表评论时：
- 当前评论所属文章从路由或接口路径中获取
- 当前评论作者从认证上下文获取
- 不允许前端自行传作者 ID

### 3.3 评论数维护策略

发表评论成功后：
- 后端同步将所属文章的 `commentCount + 1`
- 前端重新拉取文章详情或评论列表，保持页面数据一致

当前阶段先只处理“新增评论增加计数”，不提前实现删除评论和回滚计数。

## 4. API 设计

### 4.1 获取评论列表

`GET /api/posts/{postId}/comments`

处理逻辑：
- 校验文章存在且可见
- 返回该文章下所有 `VISIBLE` 评论
- 按创建时间升序排序，便于阅读讨论上下文

### 4.2 发表评论

`POST /api/posts/{postId}/comments`

请求体：
- `content`

处理逻辑：
- 仅允许已登录用户调用
- 校验文章存在且可见
- 校验评论内容长度
- 创建评论并绑定当前用户与文章
- 默认状态为 `VISIBLE`
- 文章 `commentCount` 同步加一
- 返回新评论结构

## 5. 返回结构设计

评论返回结构建议包含：
- `id`
- `content`
- `author`
- `createdAt`

其中 `author` 包含：
- `id`
- `username`
- `nickname`

## 6. 数据结构设计

### 6.1 后端重点对象

- `Comment`
- `CommentStatus`
- `CreateCommentRequest`
- `CommentResponse`
- `CommentAuthorResponse`

### 6.2 前端重点对象

- `CommentItem`
- `CreateCommentValues`

## 7. 前端交互设计

### 7.1 文章详情页评论区

页面加载时：
- 先拉取文章详情
- 再拉取评论列表

页面展示：
- 评论区标题
- 评论总数
- 每条评论的作者昵称、时间、正文

### 7.2 评论提交流程

- 已登录用户输入评论内容
- 提交后调用评论接口
- 成功后清空输入框
- 刷新评论列表和文章详情数据

### 7.3 未登录评论行为

- 允许看到评论输入区
- 提交时若未登录，提示先登录
- 不发送成功评论请求

## 8. 安全与边界

- 评论接口必须登录
- 评论内容在接口边界做长度校验
- 仅对 `VISIBLE` 文章开放评论读取与新增
- 不暴露评论作者密码或敏感信息

## 9. 重点实现约束

- 文档与重点注释统一使用简体中文
- 首版先实现单层评论，不做楼中楼回复
- 首版不做评论分页，先跑通主流程
- 评论区优先保证与文章详情页集成顺畅

## 10. 验证方案

### 10.1 后端验证

- 文章详情可读取评论列表
- 未登录发表评论失败
- 已登录发表评论成功
- 发表评论后文章 `commentCount` 正确增加

### 10.2 前端验证

- 文章详情页能展示评论列表
- 已登录用户可提交评论
- 提交后评论立即显示
- 未登录时评论提交被拦截并提示

### 10.3 联调验证

- 评论接口能正确识别当前登录用户
- 新增评论后文章详情页评论数同步更新
- 评论列表顺序符合预期
