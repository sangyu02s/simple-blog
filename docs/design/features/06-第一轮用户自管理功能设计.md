# 第一轮用户自管理功能设计文档

## 1. 目标

当前项目已经具备用户发帖、评论、点赞等基础互动能力，但用户还缺少对自己内容的持续管理能力。第一轮迭代的目标是补齐用户侧的“自管理闭环”，让系统从“能发内容”提升到“能管理自己的内容”。

本轮目标包括：
- 个人中心页面
- 我的文章列表
- 作者编辑文章
- 作者侧隐藏文章

## 2. 范围

### 2.1 本轮实现范围
- 获取当前用户个人信息
- 获取当前用户自己的文章列表
- 编辑自己发布的文章
- 作者侧将文章从前台隐藏
- 前端个人中心与我的文章页面
- 前端文章编辑页

### 2.2 暂不实现范围
- 个人资料修改
- 头像上传
- 作者恢复已隐藏文章的专属页
- 文章删除物理删除

## 3. 核心设计

### 3.1 用户自管理边界

作者只能管理自己发布的文章：
- 可以查看自己的文章
- 可以编辑自己的文章
- 可以隐藏自己的文章
- 不可以管理其他用户的文章

### 3.2 作者侧文章状态策略

继续复用现有文章状态：
- `PUBLISHED`
- `HIDDEN`

作者隐藏文章时：
- 文章状态改为 `HIDDEN`
- 前台首页与详情页不再展示
- 后台管理员仍然可以看到并继续管理

### 3.3 我的文章列表展示范围

我的文章列表应展示：
- 标题
- 摘要
- 状态
- 创建时间
- 点赞数
- 评论数

与首页不同：
- 我的文章列表不仅展示 `PUBLISHED`
- 也应展示作者自己隐藏过的 `HIDDEN` 文章

### 3.4 编辑权限规则

编辑文章时：
- 必须登录
- 当前用户必须是文章作者
- 只能编辑自己文章
- 编辑后可更新标题、摘要、正文、标签

## 4. API 设计

### 4.1 获取我的文章列表

`GET /api/me/posts`

处理逻辑：
- 从认证上下文中获取当前用户
- 返回该用户的全部文章
- 按创建时间倒序排列

### 4.2 编辑我的文章

`PUT /api/me/posts/{postId}`

请求体：
- `title`
- `summary`
- `content`
- `tags`

处理逻辑：
- 校验当前用户是否为作者
- 校验文章是否存在
- 更新文章正文与标签关系
- 保留原有文章状态

### 4.3 隐藏我的文章

`DELETE /api/me/posts/{postId}`

说明：
- 当前阶段不做物理删除
- 这里的删除语义等价于“作者隐藏文章”

处理逻辑：
- 校验当前用户是否为作者
- 将文章状态改为 `HIDDEN`
- 返回更新后的文章摘要信息

## 5. 前端页面设计

### 5.1 个人中心页

建议路由：
- `/me`

展示内容：
- 当前用户昵称
- 用户名
- 角色
- 状态
- 我的文章入口

### 5.2 我的文章页

建议路由：
- `/me/posts`

展示内容：
- 我的文章列表
- 每篇文章显示标题、摘要、状态、统计字段
- 提供“编辑”按钮
- 提供“隐藏文章”按钮

### 5.3 文章编辑页

建议路由：
- `/me/posts/:postId/edit`

展示内容：
- 标题输入框
- 摘要输入框
- 标签输入框
- 正文输入框
- 保存按钮

## 6. 数据结构设计

### 6.1 后端重点对象
- `MyPostSummaryResponse`
- `UpdateMyPostRequest`
- `MyPostService`
- `MyPostController`
- `MyProfileResponse`

### 6.2 前端重点对象
- `MyPostItem`
- `MyProfile`
- `UpdateMyPostValues`

## 7. 权限与安全

- 所有 `/api/me/**` 接口必须登录
- 编辑与隐藏文章必须校验作者身份
- 不能通过前端传作者 ID 决定归属
- 标签更新时继续复用现有标签解析规则

## 8. 验证方案

### 8.1 后端验证
- 已登录用户可获取自己的文章列表
- 已登录用户可编辑自己的文章
- 已登录用户不可编辑他人文章
- 已登录用户可隐藏自己的文章
- 隐藏后前台不再显示该文章

### 8.2 前端验证
- 个人中心页可展示当前用户信息
- 我的文章页可展示真实文章列表
- 编辑后文章详情页内容更新
- 隐藏后我的文章状态变更为 `HIDDEN`

## 9. 结论

第一轮用户自管理功能的重点不是新增新的内容类型，而是把当前内容闭环补完整。完成这一轮后，用户侧会从“可以发帖”提升到“可以持续经营自己的内容”。
