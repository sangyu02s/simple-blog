# 第二轮内容浏览增强设计文档

## 1. 目标

当前项目已经具备基础发布与互动能力，也完成了第一轮用户自管理功能。但在内容浏览体验上仍有几个明显缺口：

- 标签目前只是展示字段，不能参与内容导航
- 首页文章列表仍是全量展示，不适合内容规模增长后的浏览
- 缺少可切换的排序方式，用户只能按默认顺序查看内容

第二轮的目标是补齐“内容发现与浏览增强”能力，使文章内容更易查找、更易筛选，也更接近真实博客社区的使用体验。

本轮目标包括：
- 标签聚合页
- 文章分页
- 文章排序能力

## 2. 范围

### 2.1 本轮实现范围
- 标签列表接口
- 标签文章聚合接口
- 首页文章分页
- 标签文章分页
- 首页文章排序
- 标签文章排序
- 前端标签聚合页
- 前端分页与排序控件

### 2.2 暂不实现范围
- 关键字全文搜索
- 标签模糊搜索
- 推荐算法
- 热度分值模型

## 3. 核心设计

### 3.1 标签聚合能力

当前每篇文章已经有标签信息，因此本轮主要把标签从“展示字段”提升为“导航入口”。

用户可通过：
- 查看全部标签
- 点击某个标签
- 进入该标签下的文章聚合页

### 3.2 分页策略

首页和标签页都采用分页：
- 后端使用标准页码参数
- 前端显示当前页与翻页按钮

当前阶段建议：
- 每页固定大小，例如 5 篇文章
- 暂不开放用户自定义 page size

### 3.3 排序策略

建议支持以下排序方式：
- `latest`：最新发布
- `mostLiked`：点赞最多
- `mostCommented`：评论最多

默认排序：
- 首页默认 `latest`
- 标签页默认 `latest`

## 4. API 设计

### 4.1 获取标签列表

`GET /api/tags`

返回内容：
- 标签名
- 标签 slug

### 4.2 获取标签文章列表

`GET /api/tags/{slug}/posts?page=0&size=5&sort=latest`

处理逻辑：
- 只返回 `PUBLISHED` 文章
- 只返回包含该标签的文章
- 支持分页与排序

### 4.3 获取首页文章分页列表

`GET /api/posts?page=0&size=5&sort=latest`

处理逻辑：
- 只返回 `PUBLISHED` 文章
- 支持分页与排序

## 5. 返回结构设计

建议对文章列表接口统一使用分页结构：
- `content`
- `page`
- `size`
- `totalElements`
- `totalPages`
- `hasNext`
- `hasPrevious`

标签页和首页都可复用该结构。

## 6. 前端页面设计

### 6.1 首页增强

首页增加：
- 排序切换
- 分页按钮

### 6.2 标签聚合页

建议路由：
- `/tags/:slug`

页面展示：
- 当前标签名
- 文章列表
- 排序控件
- 分页按钮

### 6.3 标签入口

建议在以下位置可进入标签页：
- 首页文章卡片中的标签
- 文章详情页中的标签

## 7. 数据结构设计

### 7.1 后端重点对象
- `TagResponse`
- `PagedResponse<T>`
- `PostSortOption`

### 7.2 前端重点对象
- `TagItem`
- `PagedResult<T>`
- `PostSortOption`

## 8. 权限与边界

- 标签列表与标签文章聚合页可匿名访问
- 只返回 `PUBLISHED` 文章
- 标签不存在时应返回清晰错误提示

## 9. 验证方案

### 9.1 后端验证
- 标签列表接口可返回所有标签
- 标签聚合接口只返回该标签下文章
- 首页文章接口支持分页
- 标签页文章接口支持分页
- 最新、点赞最多、评论最多排序正确生效

### 9.2 前端验证
- 首页可切换排序并翻页
- 点击标签可进入标签页
- 标签页可切换排序并翻页
- 标签页为空时有明确提示

## 10. 结论

第二轮内容浏览增强的重点不是新增内容，而是提升已有内容的可发现性与可浏览性。完成这一轮后，项目会从“能读内容”进一步提升到“更方便地找到想读的内容”。
