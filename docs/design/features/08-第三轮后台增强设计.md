# 第三轮后台增强设计文档

## 1. 目标

当前后台管理模块已经具备文章、评论、用户三类基础管理能力，但随着数据规模增长，后台仍存在两个明显问题：

- 缺少筛选与搜索，列表一多就难以定位目标内容
- 缺少概览信息，管理员进入后台后无法快速把握整体状态

第三轮的目标是增强后台可管理性和可观察性，使后台从“可操作”进一步提升到“更高效地管理”。

本轮目标包括：
- 后台筛选与搜索
- 后台首页概览

## 2. 范围

### 2.1 本轮实现范围
- 后台文章按状态筛选
- 后台文章按标题搜索
- 后台评论按状态筛选
- 后台评论按所属文章标题 / 评论内容搜索
- 后台用户按状态筛选
- 后台用户按用户名 / 昵称搜索
- 后台首页统计概览接口
- 前端后台筛选输入与概览展示

### 2.2 暂不实现范围
- 复杂组合筛选
- 多条件排序器
- 导出数据
- 图表可视化

## 3. 核心设计

### 3.1 筛选策略

后台列表按最常见管理维度提供基础筛选：

#### 文章
- 状态：全部 / `PUBLISHED` / `HIDDEN`
- 关键字：标题包含

#### 评论
- 状态：全部 / `VISIBLE` / `HIDDEN`
- 关键字：文章标题或评论内容包含

#### 用户
- 状态：全部 / `ACTIVE` / `DISABLED`
- 关键字：用户名或昵称包含

### 3.2 搜索策略

当前阶段使用简单模糊匹配：
- 使用 `contains` / `like` 即可
- 不引入全文检索
- 不区分大小写优先

### 3.3 概览指标

后台首页建议展示：
- 用户总数
- 文章总数
- 评论总数
- 隐藏文章数
- 隐藏评论数
- 禁用用户数

这些指标足以让管理员快速了解当前系统状态。

## 4. API 设计

### 4.1 后台文章筛选接口

`GET /api/admin/posts?status=&keyword=`

说明：
- `status` 为空时表示全部
- `keyword` 为空时表示不过滤标题

### 4.2 后台评论筛选接口

`GET /api/admin/comments?status=&keyword=`

说明：
- `status` 为空时表示全部
- `keyword` 用于匹配文章标题或评论正文

### 4.3 后台用户筛选接口

`GET /api/admin/users?status=&keyword=`

说明：
- `status` 为空时表示全部
- `keyword` 用于匹配用户名或昵称

### 4.4 后台概览接口

`GET /api/admin/overview`

返回内容：
- `totalUsers`
- `totalPosts`
- `totalComments`
- `hiddenPosts`
- `hiddenComments`
- `disabledUsers`

## 5. 前端交互设计

### 5.1 后台概览区

后台页顶部增加概览卡片区：
- 用户总数
- 文章总数
- 评论总数
- 隐藏文章数
- 隐藏评论数
- 禁用用户数

### 5.2 列表筛选区

在文章、评论、用户三个管理区块中增加：
- 状态下拉框
- 关键字输入框
- 列表自动刷新或按钮触发刷新

### 5.3 用户体验要求

- 筛选条件变化后应即时更新列表
- 筛选为空时恢复全部数据
- 关键字为空时恢复完整结果

## 6. 数据结构设计

### 6.1 后端重点对象
- `AdminOverviewResponse`
- 文章/评论/用户仓储筛选方法

### 6.2 前端重点对象
- `AdminOverview`
- 列表筛选状态对象

## 7. 验证方案

### 7.1 后端验证
- 文章按状态筛选正确
- 文章按标题关键字搜索正确
- 评论按状态筛选正确
- 评论按关键字搜索正确
- 用户按状态筛选正确
- 用户按关键字搜索正确
- 概览接口统计值正确

### 7.2 前端验证
- 后台概览卡片可正常展示
- 切换筛选条件后列表正确变化
- 输入关键字后列表正确收缩
- 清空筛选后列表恢复完整结果

## 8. 结论

第三轮后台增强的重点是提升后台的管理效率和信息密度。完成这一轮后，后台将从“能做操作”提升到“能快速定位问题并掌握系统状态”。
