# 演示数据设计文档

## 1. 背景

当前项目使用 H2 内存数据库，本地后端重启后，注册用户、文章、评论、点赞等数据都会丢失。为了让项目在开发阶段更适合演示、联调和验收，需要在系统启动时自动预置一批稳定的示例数据。

项目早期已经通过 `DemoDataInitializer` 实现了演示用户和文章的自动初始化，但如果把长篇文章正文直接硬编码在 Java 代码里，会带来以下问题：

- 内容与初始化逻辑耦合在一起
- 长文本阅读与维护困难
- 后续修改文章内容时需要频繁改动源码
- 难以把内容资源视为独立的演示资产管理

因此，演示数据设计不仅要解决“启动即有数据”的问题，也要解决“内容如何长期维护”的问题。

## 2. 目标

本次设计目标包括：
- 预置多个普通用户账号
- 预置多篇示例文章
- 文章主题围绕中文优美散文、英文语言小故事、英文著名诗歌相关内容展开
- 数据在本地开发环境中自动初始化
- 将预置用户与文章内容资源化，降低 Java 代码中的长文本负担

## 3. 设计原则

### 3.1 仅用于开发演示

预置数据属于开发与演示环境辅助能力，不应影响未来生产环境。

### 3.2 启动即可可用

系统启动后，首页、文章详情页、登录、评论、点赞等功能应直接具备可演示的数据基础，无需手工再创建内容。

### 3.3 数据风格自然可读

预置文章内容需要尽量接近真实博客社区中的内容，而不是简单占位文本。

### 3.4 初始化逻辑与内容解耦

Java 代码负责“如何初始化”，资源文件负责“展示什么内容”，避免把长篇内容直接埋入源码。

## 4. 采用方案

本次采用：
- **Spring Boot 启动时的 DemoDataInitializer** 作为初始化入口
- `users.json`：存储预置用户信息
- `posts.json`：存储文章元信息
- `articles/*.md`：存储文章正文

这是“结构信息 + 正文内容”分离的方式，适合当前博客项目。

采用该方案的原因：
- 相比 SQL 脚本，可读性更好
- 相比硬编码正文，更易维护
- 更方便控制初始化顺序与实体关联关系
- 后续若扩展评论、点赞、后台用户等数据，也更容易复用

## 5. 初始化策略

### 5.1 用户初始化策略

系统启动时：
- 如果用户表为空，则插入预置普通用户
- 如果用户表已有数据，则不重复初始化

### 5.2 文章初始化策略

系统启动时：
- 如果文章表为空，则插入预置文章
- 如果文章表已有数据，则不重复初始化

## 6. 资源目录结构

建议结构如下：

```text
backend/src/main/resources/demo-data/
├── users.json
├── posts.json
└── articles/
    ├── 01-spring-breeze.md
    ├── 02-rainy-night.md
    ├── 03-small-lamp.md
    ├── 04-last-leaf.md
    ├── 05-frost-reading.md
    └── 06-sunset-road.md
```

## 7. 预置用户设计

建议预置 4 个普通用户：

| 用户名 | 昵称 | 角色 | 状态 |
|---|---|---|---|
| linxi | 林溪 | USER | ACTIVE |
| muyu | 暮雨 | USER | ACTIVE |
| claire | Claire | USER | ACTIVE |
| ethan | Ethan | USER | ACTIVE |

统一规则：
- 角色：`USER`
- 状态：`ACTIVE`
- 密码采用统一开发密码，便于手工测试

## 8. 预置文章主题设计

### 8.1 中文优美散文

围绕以下主题预置若干篇：
- 春日清晨
- 雨夜与旧时光
- 山川与远行
- 人生与成长

特点：
- 语言柔和
- 节奏舒缓
- 适合作为博客社区中的文学类内容展示

### 8.2 英文语言小故事

围绕以下风格预置若干篇：
- 温暖的小故事
- 带有寓意的成长故事
- 简洁但具有启发性的英文短文

特点：
- 英文自然可读
- 长度适中
- 适合作为英语学习或阅读分享内容

### 8.3 英文著名诗歌相关内容

围绕公共领域诗歌或原创整理内容预置若干篇：
- 诗歌背景介绍
- 节选赏析
- 阅读感受

说明：
- 优先采用公共领域作品相关内容
- 避免使用存在版权风险的现代诗歌全文
- 更适合用“介绍 + 节选 + 感受”的博客表达方式

## 9. 数据拆分策略

### 9.1 用户资源

`users.json` 中存放：
- 用户名
- 昵称
- 默认密码

### 9.2 文章元信息资源

`posts.json` 中存放：
- 标题
- 摘要
- 作者用户名
- 标签列表
- 正文文件路径

### 9.3 正文内容资源

Markdown 文件中只保留正文内容，不承担数据库字段以外的业务逻辑。

这样可以让：
- 代码只负责“怎么加载”
- 资源文件负责“内容是什么”

## 10. 数据关联策略

- 文章作者分散到不同用户，避免所有文章都由同一账号发布
- 标签按主题分配，例如：
  - `散文`
  - `英文故事`
  - `诗歌`
  - `阅读`
  - `生活`
- 所有预置文章默认状态为 `PUBLISHED`
- 初始 `likeCount`、`commentCount`、`viewCount` 先置 0

## 11. 初始化器职责

重构后的 `DemoDataInitializer` 负责：
1. 读取 `users.json`
2. 创建用户
3. 读取 `posts.json`
4. 根据 `contentFile` 读取对应 Markdown 正文
5. 创建文章与标签关系

它不再直接内嵌长文本内容。

## 12. 优点

### 12.1 启动即有演示数据

系统启动后可以直接演示：
- 登录
- 首页文章列表
- 文章详情
- 评论
- 点赞

### 12.2 更易维护

- 修改文案不需要改大量 Java 代码
- Markdown 对长正文更友好

### 12.3 更清晰的职责边界

- Java 负责初始化逻辑
- JSON / Markdown 负责内容资源

### 12.4 更适合后续扩展

后续若继续增加：
- 更多演示文章
- 演示评论
- 演示点赞
- 多语言演示内容

都更容易维护。

## 13. 影响范围

预计涉及：
- `DemoDataInitializer`
- 新增资源文件目录 `demo-data/`
- 新增 JSON 结构映射对象
- 新增资源读取逻辑
- `UserRepository`
- `PostRepository`
- `TagRepository`
- 密码加密能力

## 14. 验证方案

### 14.1 启动验证
- 启动后端后能自动创建预置用户
- 启动后端后能自动创建预置文章
- 重复启动时不会无限重复插入

### 14.2 内容验证
- 首页能展示预置文章
- 文章详情页能展示 Markdown 中的完整正文内容
- 作者、标签、摘要等元信息正确对应

### 14.3 登录验证
- 预置用户可以正常登录

### 14.4 联调验证
- 预置用户可登录
- 预置文章可被评论和点赞
- 标签展示正常

## 15. 结论

通过演示数据初始化 + 内容资源化，项目可以在保持“启动即有演示内容”的同时，获得更好的可维护性和可扩展性。这种方式也更符合博客项目对内容资源的长期组织需求。
